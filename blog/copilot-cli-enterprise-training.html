01 Overview

What Is Copilot CLI?

GitHub Copilot CLI is a terminal-native AI coding agent. It brings the full power of Copilot directly to your command line — planning, building, debugging, and shipping code through natural language, without ever leaving your shell.

⚡ Agentic

Plans complex tasks, executes multi-step workflows, edits files, runs tests, and iterates autonomously until the job is done.

🔗 GitHub-Native

Access repositories, issues, and pull requests using natural language — authenticated with your existing GitHub account.

🧠 Multi-Model

Choose from models by Anthropic, Google, and OpenAI. Use /model to switch or /fleet to run multiple models in parallel.

ℹ️
Plan Availability: Copilot CLI is included with all GitHub Copilot plans — Free, Pro, Pro+, Business, and Enterprise. No additional billing is required for Business and Enterprise subscribers; however, an organization admin must enable it from the Policies page.

03 Administrator Setup

Enterprise Admin Configuration

Before developers can use Copilot CLI, an enterprise or organization admin must enable it and configure policies.

Verify Copilot Subscription

Navigate to your Enterprise settings on GitHub.com. Go to Settings → Getting Started and confirm your payment method is active with a Copilot Business or Enterprise plan.

Enable Copilot CLI Policy

Go to AI Controls → Copilot. In the "Copilot Clients" section, locate Copilot CLI and set the policy to "Enabled everywhere" or "Let organizations decide." If set to "Disabled everywhere," no users can access the CLI.

Configure AI Model Access

Users can only access models enabled at the enterprise level. Enable preferred models (Claude Sonnet 4, GPT-5, Gemini 3 Pro, etc.) in AI Controls. Developers use /model to see available models.

Assign Copilot Seats

Ensure users have an assigned Copilot seat from an organization within the enterprise. Consider a self-service model where developers can claim licenses without approval for faster adoption.

Configure Network Allowlists

Add required domains to your corporate firewall (see Network Configuration below). Install custom SSL certificates for proxy inspection if applicable.

⚠️
Audit Trail: Updates to enterprise policies that affect Copilot CLI are recorded in the enterprise audit log. Model selections, MCP server policies, and Copilot CLI enable/disable actions are all auditable events.
04 Installation

Installing Copilot CLI

Multiple installation methods are supported across macOS, Linux, and Windows. Node.js v22 or later is a prerequisite for all npm-based installations.

📦 npm (All Platforms)

bash
# Prerequisite: Node.js 22+
npm install -g @github/copilot

# Launch
copilot

🍺 Homebrew (macOS/Linux)

bash
brew install github/copilot-cli/copilot-cli

# Launch
copilot

🪟 WinGet (Windows)

powershell
winget install GitHub.CopilotCLI

# Launch
copilot

📜 Install Script (macOS/Linux)

bash
# Quick install script
curl -fsSL \
  https://cli.github.com/copilot/install.sh \
  | bash

# Install as root to /usr/local/bin
curl -fsSL \
  https://cli.github.com/copilot/install.sh \
  | sudo bash
💡
Custom Install Path: When using the install script, set PREFIX to install to a custom directory. Defaults to /usr/local when run as root or $HOME/.local for non-root users. Set VERSION to pin a specific release.
05 Authentication

Authentication Methods

Two primary authentication methods are available — interactive OAuth login or Personal Access Token (PAT).

🔑 Interactive Login (Recommended)

bash
# Start Copilot CLI
copilot

# Inside the session, run:
/login

# Follow the on-screen instructions
# to authenticate via browser

Best for individual developer onboarding. Opens a browser-based OAuth flow for seamless authentication.

🎫 Personal Access Token (PAT)

bash
# Create a fine-grained PAT at:
# github.com/settings/personal-access-tokens/new
# Enable "Copilot Requests" permission

# Set the token (checked in this order):
export COPILOT_GITHUB_TOKEN="ghp_xxxxx"
# OR
export GH_TOKEN="ghp_xxxxx"
# OR
export GITHUB_TOKEN="ghp_xxxxx"

Best for CI/CD pipelines and scripted/programmatic usage. Token variables are checked in the order listed above.

🔒
Security Note: Never commit PATs to source control. Store tokens in a secrets manager or environment-specific configuration. Rotate tokens regularly according to your enterprise's security policy.

Firewall & Proxy Setup

Enterprise networks require specific domain allowlisting and proxy configuration for Copilot CLI to communicate with GitHub's services.

🌐 Proxy Environment Variables

bash
# Standard proxy configuration
export HTTP_PROXY="http://proxy.company.com:8080"
export HTTPS_PROXY="http://proxy.company.com:8080"

# For corporate SSL inspection — point to CA bundle
export NODE_EXTRA_CA_CERTS="/path/to/corporate-ca-bundle.pem"

# Verify SSL inspection status
openssl s_client -connect api.github.com:443 \
  -proxy proxy.company.com:8080 2>/dev/null \
  | openssl x509 -noout -issuer
Network Security Layers
🧱
Firewall
Domain allowlisting on port 443 for GitHub endpoints
🔀
HTTP Proxy
Basic or Kerberos auth; route traffic through corporate proxy
🔐
SSL Inspection
Corporate CA certificates for MITM proxy inspection
🛡️
WebSocket
Ensure WS connections are allowed to GitHub endpoints
⚠️
Kerberos Auth: Copilot supports Kerberos proxy authentication. Ensure the appropriate krb5 library is installed and an active ticket exists for the proxy service. This provides stronger security than basic auth by avoiding unencrypted credentials.
07 Configuration

CLI Configuration & Permissions

Configure trusted directories, tool permissions, and path access to align with your organization's security posture.

📁 Trusted Directories

On first launch, you're prompted to trust the current directory. Configuration is stored in ~/.copilot/config.json.

json
{
  "trusted_folders": [
    "/home/dev/projects",
    "/opt/company/repos"
  ]
}

Override the config location with the COPILOT_HOME environment variable.

🔧 Tool Permissions

Copilot asks approval before using tools like touch, chmod, node, or sed. Control with flags:

bash
# Interactive mode (recommended)
copilot

# Allow all tools (use cautiously!)
copilot --allow-all-tools

# Allow all paths
copilot --allow-all-paths

# Programmatic mode with auto-approve
copilot -p "fix the tests" \
  --allow-all-tools

🧩 LSP Server Configuration

Copilot CLI supports Language Server Protocol for enhanced code intelligence. LSP servers must be installed separately and configured at user or repo level:

json — ~/.copilot/lsp-config.json
{
  "servers": {
    "typescript": {
      "command": "typescript-language-server",
      "args": ["--stdio"]
    },
    "python": {
      "command": "pylsp",
      "args": []
    }
  }
}

10 Security & Governance

Enterprise Security Considerations

Critical security controls and governance practices for deploying Copilot CLI across your organization.

Enterprise Security Framework
🔐
Data Privacy
Business/Enterprise data is not used to train models. Prompts & suggestions are not retained.
📋
Audit Logging
All policy changes and model access events recorded in enterprise audit log.
🚫
Content Exclusions
Use .copilotignore and repo-level exclusions for sensitive files and paths.
🔍
Duplicate Detection
Built-in filter prevents suggestions that closely match existing public code.
🧱
Agent Firewall
Cloud agent includes built-in firewall with configurable allowlists for network access.

⚠️ Best Practices

  • Integrate secret scanning tools (e.g., GitGuardian, TruffleHog) into your pipeline
  • Exclude sensitive repositories from AI tool access entirely
  • Configure .copilotignore files for proprietary code paths
  • Require code review for all AI-generated changes
  • Set the CLI to interactive mode (not --allow-all-tools) for sensitive repos

✅ Data Retention (Biz/Enterprise)

  • Prompts & Suggestions: Not retained
  • User Engagement Data: Kept for 2 years
  • Feedback Data: Stored as needed
  • Cloud Agent Logs: Retained for account lifetime
  • Third-party model providers do not receive training data
11 Rollout Strategy

Recommended Rollout Timeline

A phased approach for enterprise-wide Copilot CLI adoption.

Week 1–2: Infrastructure Preparation

Configure firewall allowlists, proxy settings, and SSL certificates. Enable Copilot CLI policy at the enterprise level. Set up audit logging and review data retention policies.

Week 3: Pilot Team (5–10 Developers)

Select early adopters from different technology stacks. Install Copilot CLI, run authentication, and validate network connectivity. Establish custom instructions in a shared repository.

Week 4–5: Feedback & Customization

Collect feedback from the pilot team. Build team-specific agents, skills, and instructions. Install useful plugins from the Awesome Copilot marketplace. Document internal best practices.

Week 6–8: Broader Rollout

Extend access to the entire engineering organization. Deliver training sessions using this guide. Set up self-service license claiming. Monitor usage via audit logs and premium request quotas.

Ongoing: Optimization

Track ROI metrics using the Impact Reporter plugin. Iterate on custom agents and skills. Keep up with /changelog for new features. Contribute back to the Awesome Copilot community.
