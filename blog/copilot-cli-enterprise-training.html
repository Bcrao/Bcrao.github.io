Section 01

What Is Agentic Coding?

Agentic coding is a shift from AI-as-autocomplete to AI-as-collaborator. Instead of suggesting the next line, an agentic system takes a task description, plans its approach, reads your codebase for context, executes multi-step changes across files, runs tests to verify its work, and iterates until the job is done.

The developer stays in control — reviewing plans before execution, approving or denying tool use, and setting guardrails through configuration — but the agent handles the coordination.

GitHub Copilot operates in agentic mode across two surfaces:

⌨
Copilot CLI

Runs locally in your terminal. Synchronous, interactive sessions. Supports Plan mode (analyze first), Autopilot mode (work until done), and dynamic sub-agent delegation.

☁
Cloud Coding Agent

Runs asynchronously via GitHub Actions. Assign an issue, the agent spins up a VM, clones the repo, works in the background, and opens a draft PR when finished.

💡

Copilot CLI is included with all paid Copilot plans (Pro, Pro+, Business, Enterprise) at no additional cost. Your org admin needs to enable it in the organization's Copilot policies.

Section 03

Phase 1 — Enterprise Admin Setup

Before developers touch anything, an enterprise or org admin configures Copilot policies on GitHub.com. This is the control plane for everything that follows.

3.1 Enable Copilot for the Organization

Step 1: Navigate to your enterprise settings on GitHub.com → Settings → Getting Started → verify payment method.

Step 2: Go to the AI Controls tab. Enable Copilot for relevant organizations. Choose Copilot Business or Enterprise tier per org.

Step 3: Under each org, go to Settings → Policies → Copilot. Enable features: Copilot CLI, code completion, Copilot Chat, code review.

⚠

Copilot CLI is a separate toggle. Even if Copilot is enabled for your org, the CLI must be explicitly enabled in the organization's Copilot policies.

3.2 Configure Model Access

Copilot CLI supports models from Anthropic (Claude Sonnet 4.6, Claude Opus 4.6), OpenAI (GPT-5, GPT-5 mini, GPT-4.1), and Google (Gemini 3 Pro). Org admins restrict available models through the model policy. The CLI filters blocked models from the /model picker automatically.

3.3 Configure MCP Server Access

MCP (Model Context Protocol) servers let agents connect to external tools and data. Enterprise admins control access through the MCP allowlist policy.

MCP Configuration Scopes
ENTERPRISE (AI Controls → MCP)
ORGANIZATION (Policies → Copilot)
USER-LEVEL
~/.copilot/mcp-config.json ~/.copilot/config.json
WORKSPACE-LEVEL
.mcp.json .vscode/mcp.json devcontainer.json
Figure 3 — Enterprise policies override org-level settings. "Registry Only" mode blocks all unlisted servers.

Two modes: Allow all (default) lets developers use any MCP server. Registry only restricts to your approved registry. For hi-tech enterprise teams handling proprietary code, Registry-only is recommended.

3.4 Network & Firewall

Add required Copilot URLs to your allowlist (see the Copilot Allowlist Reference). The CLI communicates over HTTPS and uses localhost for LSP integration. For GHE Server, set GH_HOST.

3.5 License Assignment

Assign licenses directly to users/teams, or enable self-serve where developers claim a license without approval. Many successful enterprise rollouts use the self-serve model to reduce friction.

Phase 2 — Developer Workstation Setup

4.1 Prerequisites

Node.js 22 or later is required. Verify:

node --version   # Must be ≥ 22.x

4.2 Installation

PlatformCommand
npm (all platforms)npm install -g @github/copilot
Homebrew (macOS/Linux)brew install github/copilot/copilot
WinGet (Windows)winget install GitHub.CopilotCLI
Install script (Unix)curl -fsSL https://copilot.sh | bash
GitHub CLIgh copilot (auto-installs on first run)
CodespacesPre-installed — no setup needed

4.3 Authentication

Interactive login — run /login inside a session and follow the device-code flow in your browser.

Token-based (CI/CD) — create a fine-grained PAT with the "Copilot Requests" permission:

# Token precedence: COPILOT_GITHUB_TOKEN → GH_TOKEN → GITHUB_TOKEN
export GH_TOKEN=ghp_your_token_here

# For CI/CD with credential managers:
export GITHUB_ASKPASS=/path/to/credential-helper

4.4 Trust & Permissions

On first launch in a directory, Copilot asks you to confirm trust. This is a security boundary — only trust directories where you're comfortable letting the agent read, modify, and execute files.

Tool permissions are controlled per-invocation or via flags:

FlagEffect
--allow-all-toolsAllow all tools without prompting
--allow-tool='shell(npm test)'Allow a specific shell command
--deny-tool='shell(rm)'Block a specific command (overrides allow)
--deny-tool='write'Prevent all file modifications
🛑

--deny-tool takes precedence over --allow-all-tools. Always use deny rules as a safety net, even when broadly allowing tools.

4.5 Model Selection

Run /model inside a session to see available models. Default is Claude Sonnet 4.5. Use /fleet to run multiple models in parallel and compare outputs. Org-restricted models are filtered automatically.

Section 05

Customization Primitives

The seven building blocks of enterprise Copilot configuration. Each builds on the ones below it.

5.1 Instructions — Always-On Standards

Markdown files Copilot loads automatically at every session start. They set the baseline: coding standards, naming conventions, security rules. Never invoked — always on.

FileScope
.github/copilot-instructions.mdRepository-wide (primary)
AGENTS.md / CLAUDE.mdRepository-wide (auto-discovered)
src/.copilot-instructions.mdScoped to src/ directory and below
~/.copilot/instructions.mdUser-level (all repos)
 
# Coding Standards
- Use TypeScript strict mode for all new files
- All functions must have JSDoc comments
- Never commit secrets or API keys
- Use ESLint + Prettier formatting
- Test coverage must exceed 80%

5.2 Skills — Auto-Activated Task Runbooks

A skill is a folder containing a SKILL.md plus optional scripts, templates, and reference docs. Copilot auto-discovers skills by matching your prompt to each skill's description. Unlike instructions, skills activate only when relevant and can include executable scripts.

skills/ci-debugging/
  SKILL.md            # Description + step-by-step guide
  run-diagnostics.sh  # Script the agent can execute
  examples/           # Reference examples

5.3 Custom Agents — Session Persona

An agent is a .agent.md file with YAML frontmatter defining name, tools, model, and behavior. When selected, it shapes the entire session. Agents can delegate to sub-agents, access specific MCP servers, and invoke skills.

---
name: terraform
description: Terraform IaC specialist
model: claude-sonnet-4-6
tools:
  - mcp: terraform-registry
  - mcp: github
---
# Terraform Agent
You are an IaC specialist. Always use
Azure Verified Modules when available...

5.4 Hooks — Deterministic Enforcement

Unlike instructions (which the model can ignore), a hook is a shell script that runs at a specific lifecycle event. The preToolUse hook can programmatically approve or deny tool execution. No prompt engineering overrides a non-zero exit code.

Agent Hook Lifecycle Events
sessionStart
Session begins or resumes
userPromptSubmitted
User sends a prompt
preToolUse
Most powerful — returns approve / deny / modify
postToolUse
After tool completes (auto-format, lint)
errorOccurred
On agent error
sessionEnd
Session terminates (audit logging)
Figure 4 — The six hook events. preToolUse is the enforcement gate — it can block any tool execution.
{
  "hooks": [
    {
      "event": "preToolUse",
      "script": ".github/hooks/secret-scanner.sh",
      "description": "Scan for leaked secrets"
    },
    {
      "event": "postToolUse",
      "script": ".github/hooks/auto-lint.sh",
      "description": "Auto-format after edits"
    }
  ]
}

5.5 MCP Servers — External Tools & Data

MCP gives agents access to external systems. Two servers ship by default: GitHub MCP (issues, PRs, repo search) and Playwright MCP (web page interaction, localhost only). Custom servers configured at user, workspace, or enterprise level.

{
  "servers": {
    "terraform": {
      "type": "stdio",
      "command": "npx",
      "args": ["@hashicorp/terraform-mcp"]
    },
    "internal-api": {
      "type": "sse",
      "url": "https://mcp.internal.company.com/sse"
    }
  }
}

Connection types: STDIO (local process), SSE (server-sent events), Remote OAuth (authenticated remote).

5.6 Agentic Workflows — Autonomous Background Tasks

Markdown files that compile into GitHub Actions. You write natural language instructions; a coding agent interprets and executes them. The agent runs with read-only permissions by default — write operations go through a separate gated job with scoped permissions.

Use cases for hi-tech teams: automated issue triage, CI failure analysis, documentation drift detection, test coverage improvement, daily status reports. Supported engines: Copilot CLI, Claude Code, OpenAI Codex.

5.7 Plugins — Distribution Bundles

Plugins bundle agents, skills, hooks, and MCP configs into a single installable package. The awesome-copilot repository is the default marketplace with 175+ agents, 200+ skills, and 48+ plugins.

copilot plugin install terraform@awesome-copilot

Notable enterprise plugins: Salesforce, Security & Quality, SDLC (7 agents for full lifecycle), Azure Cloud Development, Power BI, CAST Imaging.

Section 07

Security Considerations

Trusted Directories

Only trust directories where you're comfortable letting the agent modify files. Never trust system directories or directories containing secrets.

Tool Deny Rules

Use --deny-tool flags for destructive commands (rm -rf, git push --force). Deny rules always override allow rules.

MCP Allowlist

Use Registry-only mode in production environments. Enterprise policies override org-level settings.

Hook Enforcement

Use preToolUse hooks for secret scanning, input validation, and access control. Hooks are deterministic — they cannot be bypassed by prompt injection.

URL Access Control

Configure allowed_urls and denied_urls in ~/.copilot/config to control which URLs the web_fetch tool can access.

Audit Logging

Use sessionStart and sessionEnd hooks to log all Copilot activity to your SIEM. The CLI includes OpenTelemetry-compatible request IDs.

Content Exclusion

Prevent Copilot from reading sensitive files (credentials, proprietary algorithms, compliance docs) via content exclusion rules.

Network Controls

The CLI uses per-subscription API endpoints. For GHE Server, set GH_HOST. Backstop with enterprise egress controls at your firewall.
